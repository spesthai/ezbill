# EZBill 租房账单管理系统 - 技术方案

## 项目概述

EZBill 是一个面向泰国市场的租房账单管理系统，支持月租、日租、钟点房和夜市摊位等多种租赁模式。系统提供完整的账单生成、水电费计算、多语言支持和支付验证功能。

## 范围定义（MVP 边界）

### MVP 目标（第一版必须交付）
- 房东端（Web）：登录、楼盘/房间管理、创建账单、账单列表与状态、基础统计（按月汇总）
- 租客端（H5账单）：通过分享链接访问、按浏览器语言渲染、上传 Slip
- 水电费三元组互推：输入任意两个自动补全第三个；支持记忆上次读数/单价
- 支付核验：支持 Slip2go 自动核验；失败可降级为“人工确认”

### MVP 非目标（延后到后续阶段）
- 多云灾备、跨云热备切换
- ISO27001 等第三方认证与完整安全体系建设（先做必要安全控制与日志）
- 复杂的队列系统/异步流水线（MVP 以同步 + 重试 + 降级为主）
- 专业翻译协作平台接入（Crowdin/OneSky 等）
- 原生移动端 App（先 H5 与响应式 Web）

### 关键假设与约束（需要在产品与实现中保持一致）
- 计费与金额：默认币种 THB；金额统一保留 2 位小数；单价可保留 3-4 位小数
- 时间与时区：账单周期与截止时间按泰国时区（Asia/Bangkok）解释；数据库统一使用 `TIMESTAMPTZ`
- 租客身份：租客端 H5 不要求注册登录；访问权限基于“分享链接 token”（可设置过期）
- Slip 上传：允许多次上传，保留历史；最终以“verified / 人工确认”的最新状态为准

## 核心功能需求

### 1. 用户与地址管理
- 用户登录认证系统
- 集成 Thai-address 库进行泰国地址自动补全
- 创建和管理楼盘信息（Moo, Soi, Tambon, Amphoe, Province）
- 房间标签化管理（脱敏处理）

### 2. 账单管理
- 支持多种租赁模式：月租、日租、钟点房、夜市摊位
- 账单字段（核心）：周期起止（`period_start/period_end`）、截止时间（`due_at`）、计费单位与数量（`billing_unit/billing_quantity`）、租金与水电及其他费用、总额与状态
- 水电费计算：用量、单价、金额三元组互推
- 其他费用：用户可自定义添加多个其他费用项

### 3. 智能计算逻辑
- 水电费三元组计算：输入任意两个参数，自动计算第三个
- 用量 = 本次读数 - 上次读数
- 金额 = 用量 × 单价
- 单价 = 金额 ÷ 用量
- 自动记忆上次数值，简化重复输入

### 4. 多语言支持
- 操作界面：中文、英文、泰文
- H5账单页面：MVP 支持中英泰，后续可扩展到10+种语言
- 语言优先级（H5）：浏览器语言 > 账单页手动切换 > 默认语言
- 动态语言切换

### 5. 支付验证流程
- 生成H5账单链接分享给租客
- 租客上传转账凭证（Slip）
- 通过 Slip2go API 自动验证转账真实性
- 验证通过后自动更新房间状态

### 6. 数据统计与提醒
- 月度收入统计仪表盘
- 按楼盘、房间类型分类统计
- 智能提醒：根据出租频率判断是否需要提醒
- 房间列表颜色标识提醒状态

### 权限与分享链接设计（租客 H5）
- 分享链接：`/b/{share_token}` 形式，不暴露内部 `bill_id`
- Token：随机高熵（至少 128bit），可配置过期时间；支持轮换/失效（重新生成）
- 存储方式：数据库只存 `share_token_hash`（不可逆哈希），后端仅在生成时返回明文 token 一次
- 上传权限：Slip 上传接口必须校验 `share_token`，并做频率限制、文件大小限制、MIME 白名单、EXIF 清理与幂等控制
- 访问控制：租客端仅通过 Worker 读取账单展示字段与上传 Slip，不允许直连 Supabase

### Row Level Security（RLS）策略设计

Supabase 默认所有表对 `anon` / `authenticated` 角色均可访问，**必须显式开启 RLS 并编写策略**，否则任意已登录用户可读写所有数据。

#### 设计原则

- **房东端**（`authenticated` 角色）：只能读写自己名下的数据，通过 `auth.uid()` 与 `user_id` / `profiles.id` 对比鉴权。
- **租客端 H5**：不直连 Supabase，只访问 Cloudflare Worker 提供的公开接口（Worker 内部使用 `service_role` 查询/写入）。
- **服务端 Worker**：使用 Supabase `service_role` key（绕过 RLS），仅在后端执行，不暴露给前端；Worker 负责校验 `share_token`（对比 `share_token_hash`）并做脱敏输出。

#### RLS 策略 SQL

```sql
-- ============================================================
-- 开启 RLS
-- ============================================================
ALTER TABLE profiles         ENABLE ROW LEVEL SECURITY;
ALTER TABLE properties       ENABLE ROW LEVEL SECURITY;
ALTER TABLE rooms             ENABLE ROW LEVEL SECURITY;
ALTER TABLE bills             ENABLE ROW LEVEL SECURITY;
ALTER TABLE utility_readings  ENABLE ROW LEVEL SECURITY;
ALTER TABLE payments          ENABLE ROW LEVEL SECURITY;
ALTER TABLE user_settings     ENABLE ROW LEVEL SECURITY;

-- ============================================================
-- profiles：用户只能读写自己的 profile
-- ============================================================
CREATE POLICY "profiles: owner access"
  ON profiles FOR ALL
  USING (id = auth.uid())
  WITH CHECK (id = auth.uid());

-- ============================================================
-- properties：房东只能操作自己的楼盘
-- ============================================================
CREATE POLICY "properties: owner access"
  ON properties FOR ALL
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

-- ============================================================
-- rooms：房东通过 property 归属鉴权
-- ============================================================
CREATE POLICY "rooms: owner access"
  ON rooms FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM properties
      WHERE properties.id = rooms.property_id
        AND properties.user_id = auth.uid()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM properties
      WHERE properties.id = rooms.property_id
        AND properties.user_id = auth.uid()
    )
  );

-- ============================================================
-- bills：
--   authenticated（房东）：通过 room → property 鉴权
--   租客 H5：不直连 Supabase（由 Worker 代理），因此不为 anon 开放 bills 访问策略
-- ============================================================
CREATE POLICY "bills: owner access"
  ON bills FOR ALL
  TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM rooms
        JOIN properties ON properties.id = rooms.property_id
      WHERE rooms.id = bills.room_id
        AND properties.user_id = auth.uid()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM rooms
        JOIN properties ON properties.id = rooms.property_id
      WHERE rooms.id = bills.room_id
        AND properties.user_id = auth.uid()
    )
  );

-- ============================================================
-- utility_readings：房东通过 room 归属鉴权
-- ============================================================
CREATE POLICY "utility_readings: owner access"
  ON utility_readings FOR ALL
  TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM rooms
        JOIN properties ON properties.id = rooms.property_id
      WHERE rooms.id = utility_readings.room_id
        AND properties.user_id = auth.uid()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM rooms
        JOIN properties ON properties.id = rooms.property_id
      WHERE rooms.id = utility_readings.room_id
        AND properties.user_id = auth.uid()
    )
  );

-- ============================================================
-- payments：
--   authenticated（房东）：通过 bill → room → property 鉴权
--   租客 H5：不直连 Supabase（由 Worker 代理），因此不为 anon 开放 payments 访问策略
-- ============================================================
CREATE POLICY "payments: owner access"
  ON payments FOR ALL
  TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM bills
        JOIN rooms ON rooms.id = bills.room_id
        JOIN properties ON properties.id = rooms.property_id
      WHERE bills.id = payments.bill_id
        AND properties.user_id = auth.uid()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM bills
        JOIN rooms ON rooms.id = bills.room_id
        JOIN properties ON properties.id = rooms.property_id
      WHERE bills.id = payments.bill_id
        AND properties.user_id = auth.uid()
    )
  );

-- ============================================================
-- user_settings：用户只能操作自己的设置
-- ============================================================
CREATE POLICY "user_settings: owner access"
  ON user_settings FOR ALL
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());
```

#### 租客 H5 的 share_token 鉴权实现建议

租客端不使用 Supabase JWT 登录，推荐在 **Cloudflare Worker 层**完成鉴权（推荐方案，MVP 按此实现）：

```
租客请求 /b/{share_token}
       ↓
Cloudflare Worker
  1. 计算 share_token_hash，与数据库字段对比（不落库明文 token）
  2. 用 service_role key 查询 bills WHERE share_token_hash = ?
  3. 校验 share_expires_at
  4. 校验通过后，用 service_role key 代理查询/写入
  5. 返回脱敏后的账单字段给前端（不透传内部 bill_id）
```

这样 Supabase 侧只对房东 `authenticated` 开放数据访问；租客侧所有访问都必须经过 Worker，避免 `anon` 直连带来的策略复杂度与绕过风险。

## 核心业务逻辑实现

### 1. 水电费三元组计算算法

```typescript
interface UtilityCalculation {
  usage?: number;
  unitPrice?: number;
  amount?: number;
}

function round2(value: number): number {
  return Math.round(value * 100) / 100;
}

function calculateUtility(calculation: UtilityCalculation): UtilityCalculation {
  const { usage, unitPrice, amount } = calculation;
  
  if (usage !== undefined && unitPrice !== undefined) {
    // 已知用量和单价，计算金额
    return {
      usage,
      unitPrice,
      amount: round2(usage * unitPrice)
    };
  } else if (usage !== undefined && amount !== undefined) {
    // 已知用量和金额，计算单价
    if (usage === 0) throw new Error('用量为0时无法反推单价');
    return {
      usage,
      unitPrice: amount / usage,
      amount
    };
  } else if (unitPrice !== undefined && amount !== undefined) {
    // 已知单价和金额，计算用量
    if (unitPrice === 0) throw new Error('单价为0时无法反推用量');
    return {
      usage: amount / unitPrice,
      unitPrice,
      amount
    };
  }
  
  throw new Error('至少需要提供两个参数进行计算');
}
```

**补充规则（建议写入产品与实现）**
- `usage = current - previous` 若为负数需提示用户修正（抄表错误/表回零/换表），MVP 可先阻止保存负数
- 自动计算字段应允许手动覆盖，并在 UI 标注“自动计算/已手动修改”
- 舍入规则以服务端为准，前端仅展示（避免前后端计算差异）

### 3. Slip验证流程

**支付状态机（MVP 约定）**
- `uploaded`：租客已上传 Slip，等待核验
- `verifying`：后端调用第三方核验中
- `verified`：核验通过
- `rejected`：核验失败
- `manual_approved / manual_rejected`：人工确认结果（降级流程）

**账单状态推导（建议规则）**
- 若存在任一支付记录状态为 `verified` 或 `manual_approved`，则账单 `status = paid`
- 否则若 `NOW() > due_at` 且未支付，则账单 `status = overdue`
- 其他情况账单 `status = pending`

```typescript
async function verifySlip(slipImage: File, billId: string): Promise<VerificationResult> {
  // 0. 读取账单（示例：避免使用未定义的 bill 变量）
  const bill = await getBillById(billId);
  if (!bill) throw new Error('账单不存在');

  // 1. 上传到 R2（建议存 object key）
  const slipObjectKey = await uploadToR2(slipImage, billId);
  await createPaymentRecord({
    billId,
    status: 'uploaded',
    amount: bill.totalAmount,
    slipObjectKey
  });
  
  // 2. 调用 Slip2go API（失败可降级为人工确认）
  await updatePaymentStatus(billId, { paymentStatus: 'verifying' });
  const verificationResult = await slip2goApi.verify({
    slipObjectKey,
    expectedAmount: bill.totalAmount
  });
  
  // 3. 更新支付状态
  if (verificationResult.verified) {
    await updatePaymentStatus(billId, {
      billStatus: 'paid',
      paymentStatus: 'verified',
      verificationResult
    });
  } else {
    await updatePaymentStatus(billId, {
      paymentStatus: 'rejected',
      verificationResult
    });
  }
  
  return verificationResult;
}
```

### 基础设施成本详细分解

#### 1. Cloudflare 成本分析
**Cloudflare Workers**
- 免费套餐：10万次请求/天，1000个脚本
- 超出后：$0.15/百万次请求
- **预估**：初期完全免费，用户量增长后成本可控

- 包含自动CI/CD、预览部署、自定义域名

- 操作成本：$4.50/百万次操作
- 出站流量：完全免费（0出站流量费）
- **优势**：相比AWS S3节省90%以上成本

**成本计算示例**：

#### 3. Slip2go API 成本分析

## 技术风险与应对措施（详细分析）

### 1. Thai-address库集成风险

**风险详细描述**：
- Thai-address库可能数据不完整，缺少某些地区的详细地址
- 库维护不及时，无法跟上泰国地址系统的变化
- 性能问题：地址查询响应时间过长
- 依赖第三方开源项目，存在项目停止维护的风险

**应对措施**：
1. **多层地址验证**：
   ```typescript
   async function validateThaiAddress(address: PartialAddress) {
     // 第一层：Thai-address库查询
     const thaiAddressResult = await thaiAddressLib.search(address);
     
     if (thaiAddressResult.valid) {
       return thaiAddressResult;
     }
     
     // 第二层：Google Places API备选
     const googleResult = await googlePlacesApi.search(address);
     
     if (googleResult.valid) {
       // 缓存结果到本地数据库
       await cacheAddressResult(address, googleResult);
       return googleResult;
     }
     
     // 第三层：手动输入模式
     return {
       valid: true,
       requiresManualVerification: true,
       suggestedAddress: address
     };
   }
   ```

2. **本地地址缓存**：
   - 建立本地地址数据库，缓存已验证的地址
   - 减少对第三方API的依赖
   - 提高查询速度

3. **定期数据同步**：
   - 每月同步Thai-address库更新
   - 建立地址数据质量监控
   - 用户反馈机制修正错误地址

### 2. Slip2go API可靠性风险

**风险详细描述**：
- API服务中断导致支付验证失败
- 响应延迟影响用户体验
- 验证准确率波动
- 价格变动风险

**应对措施**：
1. **多级容错机制**：
   ```typescript
   class SlipVerificationService {
     private primaryProvider = new Slip2goProvider();
     private fallbackProvider = new ManualVerificationProvider();
     
     async verifySlip(slipImage: File, billId: string) {
       try {
         // 尝试主提供商
         const result = await this.primaryProvider.verify(slipImage);
         
         if (result.confidence > 0.8) {
           return result;
         }
         
         // 低置信度时使用备选方案
         return await this.fallbackProvider.verify(slipImage);
       } catch (error) {
         // 主服务失败时降级
         console.warn('Slip2go服务失败，降级到手动验证');
         return await this.fallbackProvider.verify(slipImage);
       }
     }
   }
   ```

2. **智能缓存策略**：
   - 缓存已验证的Slip特征
   - 相似Slip快速匹配
   - 减少API调用次数

3. **监控与告警**：
   - 实时监控API响应时间和成功率
   - 设置阈值告警（如响应时间>2秒，成功率<95%）
   - 自动切换备用方案
